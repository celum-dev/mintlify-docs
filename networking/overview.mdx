VirtCloud gives you a full-featured software-defined network layer that sits alongside your virtual machines. Whether you're running a single development server or a multi-tier production application spanning several regions, the same building blocks—VPCs, subnets, firewalls, and load balancers—let you design a topology that matches your security and availability requirements without managing physical infrastructure.

## Networking Components

| Component | What It Does |
|---|---|
| **VPC (Virtual Private Cloud)** | An isolated layer-3 network scoped to a single region. VMs inside a VPC communicate over private IP addresses that are never exposed to the public internet. |
| **Subnet** | A logical subdivision of a VPC's CIDR range. Use subnets to segment workloads (e.g., `10.0.1.0/24` for web servers, `10.0.2.0/24` for databases). |
| **Security Groups / Firewalls** | Stateful rule sets attached directly to VMs. Each rule specifies a protocol, port range, and allowed source or destination CIDR. Inbound traffic is denied by default; outbound traffic is allowed by default. |
| **Load Balancers** | Managed proxies that distribute HTTP, HTTPS, or TCP traffic across a pool of VMs. They expose a single public IP and forward connections to healthy backend instances. |
| **Public IPs** | Globally routable addresses assigned to VMs or load balancers. Traffic from the internet enters your infrastructure through public IPs. |
| **Private IPs** | RFC-1918 addresses assigned within a VPC. Private IPs are only reachable from other resources in the same VPC (or a peered VPC). |

## Default Networking

Every VM you create receives a **public IP address** automatically. This lets you reach a new instance immediately over SSH or HTTP without any additional configuration—useful for development and quick experiments.

For production workloads, you'll typically place VMs inside a VPC so that internal services (databases, caches, internal APIs) communicate exclusively over private IPs and are never directly reachable from the internet.

<Note>
  Private IP addresses are only available when a VM is attached to a VPC. Without VPC setup, VMs can only communicate over their public IPs.
</Note>

## VPC Architecture

A well-structured VPC deployment looks like this:

```text
┌─────────────────────────────────────────────────────┐
│  VPC  (10.0.0.0/16)                                 │
│                                                     │
│  ┌──────────────────────┐  ┌──────────────────────┐ │
│  │  Subnet A            │  │  Subnet B            │ │
│  │  10.0.1.0/24         │  │  10.0.2.0/24         │ │
│  │  (Web Tier)          │  │  (DB Tier)           │ │
│  │                      │  │                      │ │
│  │  ┌────┐  ┌────┐      │  │  ┌────┐              │ │
│  │  │VM 1│  │VM 2│ ◄────┼──┼─►│DB 1│              │ │
│  │  └────┘  └────┘      │  │  └────┘              │ │
│  │    ▲        ▲        │  │  (firewall: only      │ │
│  └────┼────────┼────────┘  │   port 5432 from     │ │
│       │        │           │   Subnet A)          │ │
│  Load Balancer (public IP) └──────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

- The **VPC** provides the outer boundary and CIDR range.
- **Subnets** subdivide that range into logical tiers.
- **VMs attach to subnets** and receive a private IP from the subnet's range.
- **Firewall rules** are applied at the VM level, controlling which protocols and source CIDRs can reach each instance.
- A **load balancer** sits at the edge, accepting public traffic and forwarding it to the VMs in the web tier.

## Common Architectures

### Single Public VM (Dev / Test)

The simplest setup: one VM with a public IP and a firewall that opens only the ports you need. No VPC required.

```text
Internet → [Public IP] → VM (firewall: 22, 80, 443)
```

This is the default for any new VM. Add firewall rules to lock down access to known IP ranges.

### Web Tier + Private DB Tier (VPC)

The most common production pattern. Public-facing web servers sit in one subnet behind a load balancer; database servers live in a second subnet with no public IP.

```text
Internet → Load Balancer (public IP)
               ↓ round-robin
         VM-web-1, VM-web-2  (Subnet A, 10.0.1.0/24)
               ↓ private IP
         VM-db-1              (Subnet B, 10.0.2.0/24)
```

Firewall rules on the DB tier accept connections only from Subnet A's CIDR, so the database is unreachable from the internet even if its firewall were misconfigured.

### Multi-Region with Separate VPCs

For geo-redundancy or data-residency requirements, deploy an identical stack in two regions, each with its own VPC. Use VPC peering (or your application's replication mechanism) to synchronize state between regions.

```text
us-east-1 VPC (10.0.0.0/16)        eu-west-1 VPC (10.1.0.0/16)
  Load Balancer                       Load Balancer
  Web VMs + DB VMs        ◄──peer──►  Web VMs + DB VMs
```

A global DNS load balancer (e.g., latency-based routing) directs users to the nearest region.
