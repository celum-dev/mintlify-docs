VirtCloud block storage volumes are persistent network-attached disks that you can create independently of any virtual machine, attach or detach on demand, and resize at any time. Backup policies let you schedule automated point-in-time copies of both volumes and virtual machines. All endpoints require Bearer token authentication and use the base URL `https://api.virtcloud.io/v1`.

---

## Volumes

### List volumes

**`GET /volumes`** — Returns `200 OK`

<ParamField query="region" type="string">
  Filter volumes to a specific region slug.
</ParamField>

<ParamField query="status" type="string">
  Filter by volume status: `available`, `attached`, `creating`, `resizing`, or `error`.
</ParamField>

<CodeGroup>
```bash cURL
curl https://api.virtcloud.io/v1/volumes \
  -H "Authorization: Bearer YOUR_API_KEY"
```

```bash cURL (filtered)
curl "https://api.virtcloud.io/v1/volumes?region=us-east-1&status=available" \
  -H "Authorization: Bearer YOUR_API_KEY"
```
</CodeGroup>

```json Response (200 OK)
{
  "data": [
    {
      "id": "vol_abc123",
      "name": "database-disk",
      "region": "us-east-1",
      "size_gb": 100,
      "volume_type": "standard",
      "status": "attached",
      "attached_to": "vm_a1b2c3d4",
      "created_at": "2024-01-15T12:00:00Z"
    }
  ],
  "meta": { "page": 1, "per_page": 20, "total": 1, "total_pages": 1 }
}
```

<ResponseField name="data" type="array">
  Array of volume objects.

  <Expandable title="Volume object fields">
    <ResponseField name="id" type="string">Unique identifier, prefixed with `vol_`.</ResponseField>
    <ResponseField name="name" type="string">Human-readable volume name.</ResponseField>
    <ResponseField name="region" type="string">Region slug where the volume resides.</ResponseField>
    <ResponseField name="size_gb" type="integer">Provisioned size in gigabytes.</ResponseField>
    <ResponseField name="volume_type" type="string">Storage tier: `standard` or `high-perf`.</ResponseField>
    <ResponseField name="status" type="string">Current volume state: `available`, `attached`, `creating`, `resizing`, or `error`.</ResponseField>
    <ResponseField name="attached_to" type="string">ID of the virtual machine the volume is attached to, or `null` when unattached.</ResponseField>
    <ResponseField name="created_at" type="string">ISO 8601 creation timestamp.</ResponseField>
  </Expandable>
</ResponseField>

### Create a volume

**`POST /volumes`** — Returns `201 Created`

<ParamField body="name" type="string" required>
  A unique name for the volume within your account.
</ParamField>

<ParamField body="region" type="string" required>
  Region where the volume will be created. Must match the region of any VM you intend to attach it to.
</ParamField>

<ParamField body="size_gb" type="integer" required>
  Provisioned size in gigabytes. Minimum is `10`; maximum is `16384`.
</ParamField>

<ParamField body="volume_type" type="string" required>
  Storage performance tier. Use `standard` for general-purpose workloads or `high-perf` for IOPS-intensive databases and caches.
</ParamField>

<CodeGroup>
```bash cURL
curl -X POST https://api.virtcloud.io/v1/volumes \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "database-disk",
    "region": "us-east-1",
    "size_gb": 100,
    "volume_type": "standard"
  }'
```
</CodeGroup>

```json Response (201 Created)
{
  "data": {
    "id": "vol_abc123",
    "name": "database-disk",
    "region": "us-east-1",
    "size_gb": 100,
    "volume_type": "standard",
    "status": "creating",
    "attached_to": null,
    "created_at": "2024-01-15T12:00:00Z"
  }
}
```

<Tip>
  The volume transitions from `creating` to `available` within a few seconds. Poll `GET /volumes/{id}` or listen for the `volume.available` webhook event before attempting to attach the volume to a VM.
</Tip>

### Delete a volume

**`DELETE /volumes/{id}`** — Returns `204 No Content`

Permanently deletes a volume and all data it contains. This operation cannot be undone.

<ParamField path="id" type="string" required>
  The ID of the volume to delete.
</ParamField>

<Warning>
  You must detach the volume from its virtual machine before deleting it. Attempting to delete an attached volume returns a `409 Conflict` error. Detach the volume first using `POST /volumes/{id}/detach`, then issue the delete request. All data on the volume is permanently destroyed upon deletion.
</Warning>

<CodeGroup>
```bash cURL
curl -X DELETE https://api.virtcloud.io/v1/volumes/vol_abc123 \
  -H "Authorization: Bearer YOUR_API_KEY"
```
</CodeGroup>

A successful request returns `204 No Content` with no response body.

### Attach a volume to a VM

**`POST /volumes/{id}/attach`** — Returns `200 OK`

Attaches a volume to a virtual machine. The volume appears as a new block device inside the VM (for example `/dev/vdb`) and must be mounted before use.

<ParamField path="id" type="string" required>
  The ID of the volume to attach.
</ParamField>

<ParamField body="virtual_machine_id" type="string" required>
  The ID of the virtual machine to attach the volume to. The VM and volume must be in the same region.
</ParamField>

<Note>
  A volume can only be attached to one virtual machine at a time. Attempting to attach an already-attached volume to a second VM returns a `409 Conflict` error. Detach it first if you need to move it.
</Note>

<CodeGroup>
```bash cURL
curl -X POST https://api.virtcloud.io/v1/volumes/vol_abc123/attach \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"virtual_machine_id": "vm_a1b2c3d4"}'
```
</CodeGroup>

```json Response (200 OK)
{
  "data": {
    "id": "vol_abc123",
    "name": "database-disk",
    "region": "us-east-1",
    "size_gb": 100,
    "volume_type": "standard",
    "status": "attached",
    "attached_to": "vm_a1b2c3d4",
    "created_at": "2024-01-15T12:00:00Z"
  }
}
```

### Detach a volume from a VM

**`POST /volumes/{id}/detach`** — Returns `200 OK`

Detaches a volume from its virtual machine. Unmount the volume within the guest OS before issuing this request to prevent data corruption.

<ParamField path="id" type="string" required>
  The ID of the volume to detach.
</ParamField>

<Warning>
  Always unmount the volume's filesystem inside the guest OS (`umount /mnt/data`) before calling this endpoint. Forcibly detaching a mounted volume can corrupt the filesystem and result in data loss.
</Warning>

<CodeGroup>
```bash cURL
curl -X POST https://api.virtcloud.io/v1/volumes/vol_abc123/detach \
  -H "Authorization: Bearer YOUR_API_KEY"
```
</CodeGroup>

```json Response (200 OK)
{
  "data": {
    "id": "vol_abc123",
    "name": "database-disk",
    "region": "us-east-1",
    "size_gb": 100,
    "volume_type": "standard",
    "status": "available",
    "attached_to": null,
    "created_at": "2024-01-15T12:00:00Z"
  }
}
```

### Resize a volume

**`POST /volumes/{id}/resize`** — Returns `200 OK`

Increases the size of a volume. Volumes can only be grown, never shrunk. After the API call succeeds, you must resize the filesystem inside the guest OS to make use of the additional space.

<ParamField path="id" type="string" required>
  The ID of the volume to resize.
</ParamField>

<ParamField body="size_gb" type="integer" required>
  The new size in gigabytes. Must be strictly greater than the current size. Maximum is `16384`.
</ParamField>

<Note>
  After resizing, you must expand the partition and filesystem inside the guest OS. For ext4 on Linux, run `resize2fs /dev/vdb` after confirming the block device reflects the new size with `lsblk`.
</Note>

<CodeGroup>
```bash cURL
curl -X POST https://api.virtcloud.io/v1/volumes/vol_abc123/resize \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"size_gb": 200}'
```
</CodeGroup>

```json Response (200 OK)
{
  "data": {
    "id": "vol_abc123",
    "name": "database-disk",
    "region": "us-east-1",
    "size_gb": 200,
    "volume_type": "standard",
    "status": "resizing",
    "attached_to": "vm_a1b2c3d4",
    "created_at": "2024-01-15T12:00:00Z"
  }
}
```

## Backup Policies

Backup policies configure automated, recurring snapshots of your virtual machines and volumes. Backups are stored in your account and can be used to restore a VM or volume via the `backup_id` parameter.

### Enable VM backups

**`POST /virtual-machines/{id}/backups`** — Returns `201 Created`

Creates a backup policy for a virtual machine. Once enabled, backups run automatically on the specified schedule.

<ParamField path="id" type="string" required>
  The ID of the virtual machine to back up.
</ParamField>

<ParamField body="schedule" type="string" required>
  How often to take a backup: `hourly`, `daily`, or `weekly`.
</ParamField>

<ParamField body="retention_days" type="integer" required>
  Number of days to retain each backup before automatic deletion. Minimum is `1`; maximum is `365`.
</ParamField>

<CodeGroup>
```bash cURL
curl -X POST https://api.virtcloud.io/v1/virtual-machines/vm_a1b2c3d4/backups \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "schedule": "daily",
    "retention_days": 7
  }'
```
</CodeGroup>

```json Response (201 Created)
{
  "data": {
    "id": "bp_t1u2v3w4",
    "resource_type": "virtual_machine",
    "resource_id": "vm_a1b2c3d4",
    "schedule": "daily",
    "retention_days": 7,
    "next_run_at": "2024-01-16T00:00:00Z",
    "created_at": "2024-01-15T13:00:00Z"
  }
}
```

<ParamField path="id" type="string" required>

</ParamField>

<CodeGroup>
```bash cURL
curl https://api.virtcloud.io/v1/virtual-machines/vm_a1b2c3d4/backups \
  -H "Authorization: Bearer YOUR_API_KEY"
```
</CodeGroup>

```json Response (200 OK)
{
  "data": [
    {
      "id": "bkp_r1s2t3u4",
      "resource_type": "virtual_machine",
      "resource_id": "vm_a1b2c3d4",
      "size_gb": 22,
      "status": "completed",
      "taken_at": "2024-01-15T00:00:00Z",
      "expires_at": "2024-01-22T00:00:00Z"
    }
  ],
  "meta": { "page": 1, "per_page": 20, "total": 1, "total_pages": 1 }
```

<ResponseField name="data" type="array">
  Array of backup objects.

  <Expandable title="Backup object fields">
    <ResponseField name="id" type="string">Unique identifier for this backup, prefixed with `bkp_`.</ResponseField>

    <ResponseField name="resource_id" type="string">ID of the VM or volume this backup was taken from.</ResponseField>
    <ResponseField name="size_gb" type="integer">Size of the backup in gigabytes.</ResponseField>
    <ResponseField name="status" type="string">Backup status: `pending`, `in_progress`, `completed`, or `failed`.</ResponseField>
    <ResponseField name="taken_at" type="string">ISO 8601 timestamp of when the backup was initiated.</ResponseField>
    <ResponseField name="expires_at" type="string">ISO 8601 timestamp after which the backup will be automatically deleted.</ResponseField>
  </Expandable>
</ResponseField>

### Disable VM backups

**`DELETE /virtual-machines/{id}/backups`** — Returns `204 No Content`

Deletes the backup policy for a virtual machine. Existing backups are retained until their individual expiration dates but no new backups are taken.

<ParamField path="id" type="string" required>
  The ID of the virtual machine whose backup policy you want to remove.
</ParamField>

<CodeGroup>
```bash cURL
curl -X DELETE https://api.virtcloud.io/v1/virtual-machines/vm_a1b2c3d4/backups \
  -H "Authorization: Bearer YOUR_API_KEY"
```
</CodeGroup>

A successful request returns `204 No Content` with no response body.

### Enable volume backups

**`POST /volumes/{id}/backups`** — Returns `201 Created`

Creates a backup policy for a volume. Volume backups are crash-consistent snapshots taken while the volume is in its current state.

<ParamField path="id" type="string" required>
  The ID of the volume to back up.
</ParamField>

<ParamField body="schedule" type="string" required>
  Backup frequency: `hourly`, `daily`, or `weekly`.
</ParamField>

<ParamField body="retention_days" type="integer" required>
  Number of days to retain each backup before automatic deletion.
</ParamField>

<Warning>
  Volume backups are crash-consistent. For application-consistent backups of a database volume, quiesce writes within the guest OS before the scheduled backup window or use a pre-backup script.
</Warning>

<CodeGroup>
```bash cURL
curl -X POST https://api.virtcloud.io/v1/volumes/vol_abc123/backups \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "schedule": "daily",
    "retention_days": 14
  }'
```
</CodeGroup>

```json Response (201 Created)
{
  "data": {
    "id": "bp_x5y6z7a8",
    "resource_type": "volume",
    "resource_id": "vol_abc123",
    "schedule": "daily",
    "retention_days": 14,
    "next_run_at": "2024-01-16T00:00:00Z",
    "created_at": "2024-01-15T13:30:00Z"
  }
}
```
