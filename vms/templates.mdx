Templates are golden images that encode a fully configured operating environment — installed packages, tuned configuration files, security hardening, and any agents or tooling your workloads depend on. Rather than scripting post-boot configuration every time you launch a VM, you bake that work into a template once and reuse it across your entire fleet. Every VM launched from the same template starts in an identical, known-good state.

Templates are built from [snapshots](/vms/snapshots), so the workflow is: configure a VM → snapshot it → promote the snapshot to a template → launch as many VMs as you need.

## Creating a Template from a Snapshot

Before you create a template, you need an existing snapshot. See [Create and Restore VM Snapshots](/vms/snapshots) if you haven't taken one yet.

<Tabs>
  <Tab title="Dashboard">
    <Steps>
      <Step title="Navigate to Templates">
        In the left sidebar, click **Templates**, then click **New Template**.
      </Step>
      <Step title="Select a source snapshot">
        Use the snapshot picker to select the snapshot you want to promote. Only snapshots with a `complete` status appear in the list.
      </Step>
      <Step title="Name and describe the template">
        Give the template a clear, versioned name (for example, `nginx-base-v1`) and add a description that records what is installed or configured. Click **Create Template** to finish.
      </Step>
    </Steps>
  </Tab>
  <Tab title="API">
    <CodeGroup>
      ```bash cURL
      curl -X POST https://api.virtcloud.io/v1/templates \
        -H "Authorization: Bearer YOUR_API_KEY" \
        -H "Content-Type: application/json" \
        -d '{
          "name": "nginx-base-v1",
          "snapshot_id": "snap_abc123",
          "description": "Ubuntu 22.04 with Nginx pre-installed"
        }'
      ```
    </CodeGroup>

    A `201 Created` response returns the new template object:

    ```json
    {
      "id": "tmpl_p9q8r7s6",
      "name": "nginx-base-v1",
      "description": "Ubuntu 22.04 with Nginx pre-installed",
      "snapshot_id": "snap_abc123",
      "visibility": "private",
      "created_at": "2024-01-17T08:00:00Z"
    }
    ```
  </Tab>
</Tabs>

<Tip>
  Before taking the snapshot you plan to use as a template, install and configure any monitoring agents, log shippers, or security tooling you want on every VM in your fleet. Doing this once at the template level is far more reliable than running post-boot scripts on each individual VM.
</Tip>

## Listing Templates

The templates list returns both your **custom (private) templates** and any **public templates** published by VirtCloud, such as hardened OS images or application stacks:

<CodeGroup>
  ```bash cURL
  curl https://api.virtcloud.io/v1/templates \
    -H "Authorization: Bearer YOUR_API_KEY"
  ```
</CodeGroup>

```json
{
  "templates": [
    {
      "id": "tmpl_p9q8r7s6",
      "name": "nginx-base-v1",
      "description": "Ubuntu 22.04 with Nginx pre-installed",
      "visibility": "private",
      "created_at": "2024-01-17T08:00:00Z"
    },
    {
      "id": "tmpl_pub_ubuntu2204",
      "name": "ubuntu-22.04-minimal",
      "description": "Official Ubuntu 22.04 LTS minimal image",
      "visibility": "public",
      "created_at": "2023-10-01T00:00:00Z"
    }
  ],
  "meta": {
    "total": 2,
    "page": 1,
    "per_page": 25
  }
}
```

Filter to your own templates using the `?visibility=private` query parameter.

## Launching a VM from a Template

To launch a VM from a template, use the standard VM create endpoint and pass `template_id` in place of `image` or `snapshot_id`:

<CodeGroup>
  ```bash cURL
  curl -X POST https://api.virtcloud.io/v1/virtual-machines \
    -H "Authorization: Bearer YOUR_API_KEY" \
    -H "Content-Type: application/json" \
    -d '{
      "name": "nginx-prod-01",
      "region": "us-east-1",
      "size": "c1.medium",
      "template_id": "tmpl_p9q8r7s6",
      "ssh_key_ids": ["sk_abc123"],
      "tags": ["production", "web"]
    }'
  ```
</CodeGroup>

The new VM boots with the exact environment captured in the template. You can launch as many VMs from the same template as your account limits allow — each is a fully independent instance.

<Note>
  You can launch template-based VMs in any region where the template is available. Custom templates are region-specific by default. If you need to launch into a different region, create a new template from a snapshot in that region.
</Note>

## Sharing Templates Within Your Organization

If your VirtCloud account belongs to an organization, you can share private templates with other members so they can use them to launch VMs without needing access to the underlying snapshot.

<Steps>
  <Step title="Open the template detail page">
    Navigate to **Templates** in the sidebar and click the template you want to share.
  </Step>
  <Step title="Open Share settings">
    Click the **Share** button on the template detail page.
  </Step>
  <Step title="Enable organization sharing">
    Toggle **Share with organization** to on. All members of your organization will immediately be able to see and use the template.
  </Step>
</Steps>

Shared templates appear in organization members' template lists with a `visibility` of `organization`. Members can launch VMs from shared templates but cannot delete or modify them — only the template owner or an organization admin can do that.

## Deleting Templates

Delete a template you no longer need to keep your template library tidy. Deleting a template does not delete the underlying snapshot.

<CodeGroup>
  ```bash cURL
  curl -X DELETE https://api.virtcloud.io/v1/templates/tmpl_p9q8r7s6 \
    -H "Authorization: Bearer YOUR_API_KEY"
  ```
</CodeGroup>

A successful deletion returns `204 No Content`. If any VMs are currently being provisioned from the template at the time of deletion, those provisioning operations will complete normally — the template is only required at launch time.

<Warning>
  Deleting a template that has been shared with your organization will immediately remove it from all members' template lists. Coordinate with your team before deleting shared templates to avoid disrupting active provisioning workflows.
</Warning>

## Best Practices

Following a few conventions when building and maintaining templates will save you significant time as your infrastructure grows.

<CardGroup cols={2}>
  <Card title="Keep base images minimal" icon="feather">
    Install only what every VM in the fleet needs — OS hardening, agents, and shared tooling. Application-specific configuration should live in user data scripts or a configuration management layer, not baked into the template.
  </Card>
  <Card title="Document what's installed" icon="file-lines">
    Use the template description field to record the OS version, key package versions, and any non-default configuration. Future you (and your teammates) will appreciate knowing exactly what is inside.
  </Card>
  <Card title="Version your template names" icon="code-branch">
    Use a naming scheme like `nginx-base-v1`, `nginx-base-v2` rather than overwriting a single name. This lets you roll back to a previous template version by name if a new one causes issues, and it gives you a clear history of changes.
  </Card>
  <Card title="Audit and retire old templates" icon="trash">
    Old template versions accumulate storage costs through their underlying snapshots. Once you have validated a new template version in production, delete the previous version's template and snapshot to clean up.
  </Card>
</CardGroup>
