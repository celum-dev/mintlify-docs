A snapshot is a complete, point-in-time copy of a virtual machine's disk state. VirtCloud stores snapshots independently from the VM, so they persist even if the original VM is deleted. Snapshots are a lightweight safety net — use them before applying OS updates, testing configuration changes, or promoting a VM to a template for fleet deployment.

Common scenarios for snapshots:

- **Before updates** — capture the current state before applying kernel updates or major package upgrades.
- **Testing changes** — snapshot before experimenting so you can roll back to a known-good state.
- **Rollback points** — mark stable application states you can return to if a deployment goes wrong.

## Creating a Snapshot

<Tabs>
  <Tab title="Dashboard">
    <Steps>
      <Step title="Open the VM detail page">
        In the left sidebar, click **Virtual Machines** and select the VM you want to snapshot.
      </Step>
      <Step title="Navigate to Snapshots">
        Click the **Snapshots** tab on the VM detail page.
      </Step>
      <Step title="Create the snapshot">
        Click **Take Snapshot**, enter a descriptive name (for example, `pre-update-backup`), and click **Confirm**. The snapshot will appear in the list with a status of **Pending**.
      </Step>
      <Step title="Wait for completion">
        The snapshot status will transition to **Complete** once the disk has been fully captured. This typically takes 1–5 minutes depending on disk size.
      </Step>
    </Steps>
  </Tab>
  <Tab title="API">
    Send a `POST` request to `/v1/snapshots` with the target VM ID and a name:

    <CodeGroup>
      ```bash cURL
      curl -X POST https://api.virtcloud.io/v1/snapshots \
        -H "Authorization: Bearer YOUR_API_KEY" \
        -H "Content-Type: application/json" \
        -d '{
          "virtual_machine_id": "vm_abc123",
          "name": "pre-update-backup"
        }'
      ```
    </CodeGroup>

    A `201 Created` response is returned immediately, with the snapshot object in a `pending` state:

    ```json
    {
      "id": "snap_xyz789",
      "name": "pre-update-backup",
      "virtual_machine_id": "vm_abc123",
      "status": "pending",
      "size_gb": null,
      "created_at": "2024-01-15T14:00:00Z"
    }
    ```

    Once the snapshot has been fully captured, the status transitions to `complete` and the `size_gb` field is populated:

    ```json
    {
      "id": "snap_xyz789",
      "name": "pre-update-backup",
      "virtual_machine_id": "vm_abc123",
      "status": "complete",
      "size_gb": 12.4,
      "created_at": "2024-01-15T14:00:00Z",
      "completed_at": "2024-01-15T14:02:37Z"
    }
    ```

    Poll `GET /v1/snapshots/snap_xyz789` to check progress.
  </Tab>
</Tabs>

<Note>
  You can take a snapshot of a running VM. VirtCloud uses a crash-consistent snapshot mechanism, which is suitable for most workloads. For database VMs, consider flushing pending writes (for example, `FLUSH TABLES WITH READ LOCK` in MySQL) before taking the snapshot to ensure full consistency.
</Note>

## Listing Snapshots

Retrieve all snapshots in your account:

<CodeGroup>
  ```bash cURL
  curl https://api.virtcloud.io/v1/snapshots \
    -H "Authorization: Bearer YOUR_API_KEY"
  ```
</CodeGroup>

```json
{
  "snapshots": [
    {
      "id": "snap_xyz789",
      "name": "pre-update-backup",
      "virtual_machine_id": "vm_abc123",
      "status": "complete",
      "size_gb": 12.4,
      "created_at": "2024-01-15T14:00:00Z"
    },
    {
      "id": "snap_def456",
      "name": "post-nginx-install",
      "virtual_machine_id": "vm_abc123",
      "status": "complete",
      "size_gb": 13.1,
      "created_at": "2024-01-16T09:15:00Z"
    }
  ],
  "meta": {
    "total": 2,
    "page": 1,
    "per_page": 25
  }
}
```

Filter snapshots by VM using the `?virtual_machine_id=vm_abc123` query parameter.

## Restoring from a Snapshot

Restoring a snapshot does not overwrite your existing VM. Instead, VirtCloud launches a **new virtual machine** pre-loaded with the snapshot's disk state. This allows you to compare the restored environment against the current one before decommissioning either.

Use the standard VM create endpoint, substituting `snapshot_id` for `image`:

<CodeGroup>
  ```bash cURL
  curl -X POST https://api.virtcloud.io/v1/virtual-machines \
    -H "Authorization: Bearer YOUR_API_KEY" \
    -H "Content-Type: application/json" \
    -d '{
      "name": "web-server-restored",
      "region": "us-east-1",
      "size": "c1.medium",
      "snapshot_id": "snap_xyz789",
      "ssh_key_ids": ["sk_abc123"]
    }'
  ```
</CodeGroup>

The new VM will boot with the exact disk state captured at snapshot time, including all installed software, configuration files, and data.

<Warning>
  The restored VM is an independent copy — changes you make to it do not affect the original VM or the snapshot. If you intend to replace the original VM, verify the restored VM is working correctly before deleting the original.
</Warning>

## Deleting Snapshots

Delete a snapshot you no longer need to stop incurring storage charges:

<CodeGroup>
  ```bash cURL
  curl -X DELETE https://api.virtcloud.io/v1/snapshots/snap_xyz789 \
    -H "Authorization: Bearer YOUR_API_KEY"
  ```
</CodeGroup>

A successful deletion returns `204 No Content`. Deleted snapshots cannot be recovered.

<Warning>
  If a snapshot has been used to create a [template](/vms/templates), you must delete the template first before you can delete the underlying snapshot.
</Warning>

## Snapshot Pricing

Snapshots are billed based on the amount of data stored, measured in GB per month. The `size_gb` field in the snapshot object reflects the compressed on-disk size — this is the value used for billing, not the raw disk size of the source VM.

You can review your current snapshot storage usage and associated charges in the **Billing** section of your account dashboard.

<Tip>
  Clean up old snapshots regularly to keep storage costs low. A practical policy is to retain only the last two or three snapshots per VM and delete older ones once you are confident a deployment is stable.
</Tip>

## Limitations

Keep the following constraints in mind when working with snapshots:

- **Maximum snapshots per VM** — you can hold up to **10 snapshots** per virtual machine. To create a new snapshot beyond this limit, delete an existing one first.
- **Full disk capture** — a snapshot captures the entire root disk state, not just the changed blocks. There is no incremental snapshot option at this time.
- **Region-bound** — snapshots are stored in the same region as the source VM. Restoring a snapshot into a different region is not currently supported.
