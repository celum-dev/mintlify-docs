title: "Configure Automated VM and Volume Backups in VirtCloud"

---

VirtCloud backups run automatically on a schedule you define and are retained for as long as your policy specifies — no manual intervention required. Unlike snapshots, which you trigger yourself at a single point in time, backups give you a continuous, rolling archive of recovery points. When something goes wrong, you can restore a VM or volume to any available backup in minutes.

## Enabling backups for a VM

<Tabs>
  <Tab title="Dashboard">
    <Steps>
      <Step title="Open VM settings">
        Navigate to **Virtual Machines**, click the VM you want to protect, and open the **Backups** tab.
      </Step>
      <Step title="Enable backups">
        Click **Enable Backups**, choose a schedule (**Hourly**, **Daily**, or **Weekly**), and set the retention period.
      </Step>
      <Step title="Save">
        Click **Save**. The first backup runs at the next scheduled interval.
      </Step>
    </Steps>
  </Tab>
  <Tab title="API">
    Send a `POST` request to `/v1/virtual-machines/{id}/backups`:

    ```bash
    curl -X POST https://api.virtcloud.io/v1/virtual-machines/vm_abc123/backups \
      -H "Authorization: Bearer YOUR_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "schedule": "daily",
        "retention_days": 7
      }'
    ```

    A successful response confirms the backup policy:

    ```json
    {
      "virtual_machine_id": "vm_abc123",
      "schedule": "daily",
      "retention_days": 7,
      "enabled": true,
      "next_run_at": "2024-06-02T00:00:00Z"
    }
    ```
  </Tab>
</Tabs>

## Available schedules

| Schedule | Frequency | Recommended for |
|---|---|---|
| `hourly` | Every hour | Mission-critical workloads requiring tight RPO |
| `daily` | Once per day (midnight UTC) | Production applications, databases |
| `weekly` | Once per week (Sunday midnight UTC) | Development and staging environments |

## Retention periods

- **Daily backups**: retain for 1 to 30 days. After the retention window expires, backups are automatically deleted.
- **Weekly backups**: retain for 1 to 52 weeks.
- **Hourly backups**: retain for 1 to 72 hours.

<Note>
  Retention is enforced on a rolling basis. If you set a 7-day retention for daily backups, you will always have the last 7 daily backups available. The oldest backup is deleted automatically when a new one is created beyond the retention limit.
</Note>

## Enabling backups for a volume

The backup API for volumes mirrors the VM backup API. Send a `POST` request to `/v1/volumes/{id}/backups`:

<CodeGroup>
  ```bash Enable volume backups
  curl -X POST https://api.virtcloud.io/v1/volumes/vol_a1b2c3d4/backups \
    -H "Authorization: Bearer YOUR_API_KEY" \
    -H "Content-Type: application/json" \
    -d '{
      "schedule": "daily",
      "retention_days": 7
    }'
  ```

  ```json Response
  {
    "volume_id": "vol_a1b2c3d4",
    "schedule": "daily",
    "retention_days": 7,
    "enabled": true,
    "next_run_at": "2024-06-02T00:00:00Z"
  }
  ```
</CodeGroup>

## Listing backups

Retrieve all backups for a specific VM:

```bash
curl https://api.virtcloud.io/v1/virtual-machines/vm_abc123/backups \
  -H "Authorization: Bearer YOUR_API_KEY"
```

Each backup object in the response includes an `id` (e.g., `bkp_xyz`), the `created_at` timestamp, `size_gb`, and `status`. Use the backup `id` when restoring.

## Restoring from a backup

You can restore a backup to a **new VM** (leaving the original untouched) or restore a volume **in place**.

<Tabs>
  <Tab title="Restore to a new VM">
    Pass `backup_id` in the VM creation request. VirtCloud provisions a new VM pre-loaded with the backup's disk image:

    ```bash
    curl -X POST https://api.virtcloud.io/v1/virtual-machines \
      -H "Authorization: Bearer YOUR_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "name": "restored-vm",
        "region": "us-east-1",
        "tier": "standard-2",
        "backup_id": "bkp_xyz"
      }'
    ```

    The original VM and its backups are not affected. This is the safest restore path for production incidents — validate the restored VM before decommissioning the original.
  </Tab>
  <Tab title="Restore a volume in place">
    Restore a volume to a previous backup state with `POST /v1/volumes/{id}/restore`:

    ```bash
    curl -X POST https://api.virtcloud.io/v1/volumes/vol_a1b2c3d4/restore \
      -H "Authorization: Bearer YOUR_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "backup_id": "bkp_xyz"
      }'
    ```

    <Warning>
      Restoring a volume **in place overwrites all current data** on that volume with the backup's contents. Detach the volume from any VM before restoring, and make sure you have a recent backup or snapshot of the current state if you might need it later.
    </Warning>
  </Tab>
</Tabs>

## Disabling backups

To stop scheduled backups for a VM, send a `DELETE` request to the backups endpoint. Existing backup copies are retained until their individual retention windows expire — they are not immediately deleted.

```bash
curl -X DELETE https://api.virtcloud.io/v1/virtual-machines/vm_abc123/backups \
  -H "Authorization: Bearer YOUR_API_KEY"
```

To disable backups and immediately purge all existing backup copies, add `"purge": true` to the request body:

```bash
curl -X DELETE https://api.virtcloud.io/v1/virtual-machines/vm_abc123/backups \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"purge": true}'
```

<Warning>
  Setting `"purge": true` permanently deletes **all existing backups** for the resource. This action cannot be undone.
</Warning>

## Backup storage cost

Backup storage is billed at **$0.05 / GB-month** based on the compressed size of each stored backup. Costs accumulate across all retained backup copies — a 100 GB VM with 7 daily backups stored for a full month would incur charges for up to 700 GB-months of backup storage, depending on compression ratios.

## Best practices

<Steps>
  <Step title="Use daily backups with 7-day retention for production workloads">
    A daily schedule with 7 days of retention gives you a full week of recovery points without excessive storage costs. For stricter recovery objectives, switch to hourly.
  </Step>
  <Step title="Enable backups on volumes separately from VMs">
    If you store your database or application data on a separate block volume, enable a backup policy on that volume directly. VM backups capture the boot disk only — they do not include attached volumes unless you configure volume backups independently.
  </Step>
  <Step title="Test your restores regularly">
    A backup you have never restored from is an untested assumption. Schedule a periodic restore drill — restore to a new VM, verify the application starts and data looks correct, then tear down the test VM.
  </Step>
  <Step title="Combine backups with snapshots for pre-change safety">
    Before a major deployment or database migration, take a manual snapshot in addition to relying on your scheduled backup. Snapshots are instant and give you a guaranteed known-good state immediately before the change.
  </Step>
</Steps>

<Tip>
  Test your restore process at least once a month in a non-production environment. Discovering that a backup is corrupt or that your restore procedure has gaps during an actual incident is far more costly than the time spent on a routine drill.
</Tip>
