---
title: "Create and Attach Persistent Block Storage Volumes"
sidebarTitle: "Volumes"
description: "Provision block storage volumes in VirtCloud and attach them to virtual machines. Resize volumes on demand and detach them to move data between VMs."
---

Block storage volumes are independent, persistent disks that you can create, attach to a VM, detach, and reattach to a different VM at any time. Unlike the boot disk, a volume is not tied to the lifecycle of any VM — it survives reboots and persists even after the VM it was attached to is deleted. This makes volumes the right place to store databases, user uploads, and any other data you want to keep separate from the operating system.

## Creating a volume

<Tabs>
  <Tab title="Dashboard">
    <Steps>
      <Step title="Open the Volumes page">
        In the VirtCloud console, navigate to **Storage** → **Volumes** in the left sidebar, then click **Create Volume**.
      </Step>
      <Step title="Configure the volume">
        Enter a name, choose a region, set the size in GB, and select a performance tier (**Standard SSD** or **High-Performance SSD**).
      </Step>
      <Step title="Create">
        Click **Create Volume**. The volume moves from **creating** to **available** within a few seconds.
      </Step>
    </Steps>
  </Tab>
  <Tab title="API">
    Send a `POST` request to `/v1/volumes` with the volume configuration:

    ```bash
    curl -X POST https://api.virtcloud.io/v1/volumes \
      -H "Authorization: Bearer YOUR_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "name": "database-disk",
        "region": "us-east-1",
        "size_gb": 100,
        "volume_type": "standard"
      }'
    ```

    A successful response returns the new volume object:

    ```json
    {
      "id": "vol_a1b2c3d4",
      "name": "database-disk",
      "status": "creating",
      "size_gb": 100,
      "volume_type": "standard",
      "region": "us-east-1",
      "created_at": "2024-06-01T10:00:00Z"
    }
    ```

    The `status` field transitions from `creating` to `available` once the volume is ready to attach. Poll `GET /v1/volumes/vol_a1b2c3d4` to check progress, or listen for the `volume.available` webhook event.
  </Tab>
</Tabs>

<Note>
  To provision a High-Performance SSD volume, set `"volume_type": "high-perf"` in the API request, or select **High-Performance SSD** in the dashboard. High-Performance SSD volumes deliver up to 16,000 IOPS compared to 3,000 IOPS for Standard SSD.
</Note>

## Attaching a volume to a VM

A volume must be in the `available` state before you can attach it. You can only attach a volume to one VM at a time.

<Tabs>
  <Tab title="Dashboard">
    <Steps>
      <Step title="Find the volume">
        Go to **Storage** → **Volumes** and click the volume you want to attach.
      </Step>
      <Step title="Attach">
        Click **Attach**, select the target VM from the dropdown, and click **Confirm**.
      </Step>
    </Steps>
  </Tab>
  <Tab title="API">
    ```bash
    curl -X POST https://api.virtcloud.io/v1/volumes/vol_a1b2c3d4/attach \
      -H "Authorization: Bearer YOUR_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "virtual_machine_id": "vm_abc"
      }'
    ```
  </Tab>
</Tabs>

After attaching, the volume appears as a raw block device inside the VM — for example, `/dev/vdb` on Linux. You need to format and mount it before use.

### Mounting the volume on Linux

<Steps>
  <Step title="Identify the device">
    Log in to your VM and run `lsblk` to confirm the device name (commonly `/dev/vdb` for the first attached volume).

    ```bash
    lsblk
    ```
  </Step>
  <Step title="Format the volume (first time only)">
    Format the device with a filesystem. Skip this step if you are reattaching a volume that already contains data.

    ```bash
    sudo mkfs.ext4 /dev/vdb
    ```
  </Step>
  <Step title="Create a mount point and mount the volume">
    ```bash
    sudo mkdir /mnt/data
    sudo mount /dev/vdb /mnt/data
    ```
  </Step>
  <Step title="Persist the mount across reboots">
    Add an entry to `/etc/fstab` so the volume is remounted automatically when the VM restarts:

    ```bash
    echo '/dev/vdb /mnt/data ext4 defaults 0 0' | sudo tee -a /etc/fstab
    ```
  </Step>
</Steps>

<Warning>
  Formatting a device with `mkfs` **destroys all existing data** on that device. Only run `mkfs.ext4` when attaching a brand-new volume for the first time.
</Warning>

## Listing volumes

Retrieve all volumes in your account:

```bash
curl https://api.virtcloud.io/v1/volumes \
  -H "Authorization: Bearer YOUR_API_KEY"
```

The response is an array of volume objects. Use the `status` field to filter by state, and `virtual_machine_id` to see which VM each volume is attached to.

## Resizing a volume

You can increase the size of any volume at any time — even while it is attached to a running VM.

```bash
curl -X POST https://api.virtcloud.io/v1/volumes/vol_a1b2c3d4/resize \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "size_gb": 200
  }'
```

<Warning>
  Volumes can only be **increased** in size — you cannot shrink a volume. Plan your initial size conservatively; you can always grow it later.
</Warning>

After the API call completes, you must extend the filesystem inside the VM to make use of the new space. For ext4:

```bash
sudo resize2fs /dev/vdb
```

## Detaching a volume

Detaching a volume removes it from the VM and returns it to the `available` state, ready to attach elsewhere.

<Warning>
  Always unmount the volume inside the OS **before** detaching it from the API or dashboard. Detaching a mounted volume can cause data corruption and filesystem errors.
</Warning>

<Steps>
  <Step title="Unmount the volume in the OS">
    ```bash
    sudo umount /mnt/data
    ```

    If you added the volume to `/etc/fstab`, remove or comment out that line to prevent boot errors after detaching.
  </Step>
  <Step title="Detach via API or dashboard">
    ```bash
    curl -X POST https://api.virtcloud.io/v1/volumes/vol_a1b2c3d4/detach \
      -H "Authorization: Bearer YOUR_API_KEY"
    ```

    Or click **Detach** on the volume detail page in the dashboard.
  </Step>
</Steps>

## Deleting a volume

Deleting a volume permanently removes all data stored on it. The volume must be in the `available` state (i.e., not attached to any VM) before you can delete it.

```bash
curl -X DELETE https://api.virtcloud.io/v1/volumes/vol_a1b2c3d4 \
  -H "Authorization: Bearer YOUR_API_KEY"
```

<Warning>
  Deleting a volume is **irreversible**. All data on the volume is permanently destroyed. If you need to preserve the data, create a snapshot or backup before deleting.
</Warning>

## Volume states

| State | Meaning |
|---|---|
| `creating` | The volume is being provisioned. Wait for `available` before attaching. |
| `available` | The volume is ready to attach to a VM. |
| `attached` | The volume is currently attached to a VM. |
| `detaching` | A detach operation is in progress. |
| `error` | An operation failed. Check the event log or contact support. |
