Before you start building on VirtCloud it's worth understanding the vocabulary the platform uses and how its resources relate to each other. This page explains every major concept — what it is, how it behaves, and when you'd use it. You don't need to read this front-to-back; treat it as a reference you can jump back to whenever something in another guide needs clarification.

## How VirtCloud Resources Relate to Each Other

At the centre of everything is the **Virtual Machine** (VM). A VM is a compute instance that runs in a specific **Region**. When you create a VM you choose an **Image** (the OS to boot from) and a **Size** (the amount of CPU, RAM, and root disk). You can then connect the VM to a **Network / VPC**, attach additional **Volumes** for persistent storage, and protect it with a **Firewall** rule set. **Snapshots** let you capture the state of a VM or volume at a point in time. **SSH Keys** are how you authenticate when you log into the VM. **Load Balancers** sit in front of a group of VMs to distribute incoming traffic.

The diagram below describes these relationships in plain terms:

```
Region
└── VPC / Network
    ├── Subnet
    │   ├── VM ──── SSH Key (for access)
    │   │    ├──── Image (boot OS)
    │   │    ├──── Root Disk (included in Size)
    │   │    ├──── Volume (attached block storage)
    │   │    └──── Firewall (inbound/outbound rules)
    │   └── Load Balancer ──► VM pool
    └── Snapshot (point-in-time copy of VM or Volume)
```

Most resources are **region-scoped**: a volume created in `us-east-1` can only be attached to a VM in `us-east-1`. Keep this in mind when planning multi-region architectures.

## Core Concepts

<AccordionGroup>

  <Accordion title="Virtual Machines (VMs)">
    A Virtual Machine is the fundamental compute unit in VirtCloud. It's an isolated server instance running on VirtCloud's hypervisor infrastructure, giving you dedicated vCPU, RAM, and disk resources that behave just like a physical machine — but provisioned in seconds.

    ### Lifecycle States

    Every VM moves through a defined set of states:

    | State | Meaning |
    |---|---|
    | `provisioning` | VirtCloud is allocating resources and booting the OS. |
    | `running` | The VM is online and accepting connections. |
    | `stopping` | A graceful shutdown has been requested and is in progress. |
    | `stopped` | The VM is offline. Resources are still reserved; billing continues. |
    | `rebooting` | The VM is restarting. It will return to `running` automatically. |
    | `resizing` | The VM is being migrated to a different size tier. |
    | `error` | Something went wrong. Check the activity log and contact support if the state persists. |

    ### What Belongs to a VM

    When you delete a VM, the following are removed along with it:
    - The root disk (the OS disk included in the VM size)
    - The public IP address (it is released back to the pool)
    - Any private IP assignments on that VM's network interface

    The following **are not** removed when you delete a VM:
    - Attached **Volumes** (persistent block storage) — they detach and persist independently.
    - **Snapshots** of the VM — they remain in your account until you delete them.
    - **Firewall** rules — firewall policies exist independently and can be re-attached to other VMs.

    <Tip>
      If you want to preserve the state of a VM before deleting or rebuilding it, take a snapshot first. You can restore from that snapshot to a new VM at any time.
    </Tip>
  </Accordion>

  <Accordion title="Regions & Availability Zones">
    A **Region** is a geographic location where VirtCloud operates data center infrastructure. Each region is an independent fault domain with its own power, networking, and cooling. Choosing the right region matters for three reasons:

    1. **Latency** — deploy close to your users for the lowest round-trip times.
    2. **Compliance** — some regulations (e.g., GDPR in Europe) require data to stay within a specific jurisdiction.
    3. **Redundancy** — running workloads in multiple regions protects against a regional outage affecting all of your services simultaneously.

    VirtCloud region identifiers follow a `<continent>-<direction>-<number>` format — for example, `us-east-1`, `eu-west-1`, `ap-southeast-1`. You can list all available regions and their current status via `GET /v1/regions`.

    Within each region, VirtCloud may offer multiple **Availability Zones** (AZs). An AZ is a physically separate facility inside the same region. Spreading VMs across AZs within a region gives you redundancy against facility-level events while keeping inter-VM latency very low.

    <Note>
      Resources are region-scoped. A volume, snapshot, or VPC created in one region cannot be directly used in another region. Use snapshot replication if you need to copy an image across regions.
    </Note>
  </Accordion>

  <Accordion title="VM Sizes">
    The **size** of a VM defines its resource allocation: virtual CPUs (vCPUs), RAM, and the size of the root disk. VirtCloud sizes are grouped into families; the `c1` family is general-purpose and the `c2` family offers higher CPU-to-RAM ratios suited to compute-intensive workloads.

    ### Standard Size Tiers

    | Size | vCPU | RAM | Root Disk | Best For |
    |---|---|---|---|---|
    | `c1.small` | 1 vCPU | 1 GB | 20 GB | Development, testing, lightweight services |
    | `c1.medium` | 2 vCPU | 4 GB | 40 GB | Web servers, small databases, staging environments |
    | `c2.large` | 4 vCPU | 8 GB | 80 GB | Application servers, moderate-traffic APIs |
    | `c2.xlarge` | 8 vCPU | 16 GB | 160 GB | High-traffic services, data processing, larger databases |

    You can resize a VM at any time through the dashboard or the API. Resizing requires a reboot and the VM will briefly pass through the `resizing` state. The root disk size can be increased but not decreased — always ensure you have a snapshot before resizing.

    <Tip>
      Not sure which size to start with? `c1.medium` is a reasonable default for most web applications. You can always scale up once you understand your actual resource usage.
    </Tip>
  </Accordion>

  <Accordion title="Images">
    An **Image** is the operating system template used to boot a VM. When you create a VM, VirtCloud copies the image onto the root disk and boots from it.

    ### Public Images

    VirtCloud maintains a library of **public images** — official, up-to-date builds of popular operating systems including Ubuntu, Debian, CentOS Stream, Rocky Linux, and Windows Server. These images are vetted and updated regularly. You don't need to do anything to use them; they're available to every account.

    ### Custom Images & Templates

    You can create your own **custom images** in two ways:

    - **From a snapshot**: Take a snapshot of a running or stopped VM, then convert it to a reusable image. Every new VM created from that image will boot with whatever OS configuration, installed packages, and files were present at snapshot time — saving you the setup time on every new deployment.
    - **Import**: Upload your own disk image (qcow2 or raw format) via the API to use as a base for new VMs.

    Custom images are private to your account (or your team, if you share them within a project). They're ideal for pre-baking application runtimes, security hardening baselines, or golden AMI-style workflows.
  </Accordion>

  <Accordion title="Networks & VPCs">
    A **Virtual Private Cloud (VPC)** is a logically isolated private network inside VirtCloud. All your VMs run inside a VPC, and by default they're placed in a default VPC automatically created for you when you first use a region.

    ### Key Networking Concepts

    - **Subnet**: A subdivision of a VPC's IP address range. You assign VMs to a subnet when you create them. Subnets can be public (routing to the internet via a gateway) or private (internal traffic only).
    - **Public IP**: An internet-routable IP address assigned to a VM so it can be reached from outside your VPC. Public IPs can be reserved and reassigned between VMs.
    - **Private IP**: An internal IP address within your VPC. VMs on the same VPC can communicate over private IPs without traffic leaving VirtCloud's network — faster and free of egress charges.
    - **Internet Gateway**: Attached to a public subnet to allow VMs in that subnet to send and receive internet traffic.

    ### Default vs. Custom VPCs

    Your default VPC is ready to use immediately — all VMs you create without specifying a network are placed here. For production environments you may want to create a custom VPC with carefully planned subnets, allowing you to isolate different tiers (e.g., web servers in a public subnet, databases in a private subnet with no direct internet access).
  </Accordion>

  <Accordion title="Volumes">
    A **Volume** is a persistent block storage device that you can attach to a VM, just like a physical hard drive. Unlike the root disk (which is part of the VM itself), a volume exists independently of any VM.

    This makes volumes the right choice for:
    - **Databases** — your data persists even if you delete or replace the VM running your database engine.
    - **Shared state** — detach a volume from one VM and attach it to another to migrate data without copying it over the network.
    - **Large datasets** — add capacity to a VM without needing to resize the whole machine.

    Volumes are formatted with the filesystem of your choice (ext4, xfs, etc.) after attachment. They can be resized upward at any time. A volume can only be attached to one VM at a time in read-write mode.

    <Note>
      Volumes are billed independently of VMs. A volume that is detached (not attached to any VM) still incurs storage charges. Delete volumes you no longer need to stop billing for them.
    </Note>
  </Accordion>

  <Accordion title="Snapshots">
    A **Snapshot** is a point-in-time, read-only copy of either a VM's root disk or an attached Volume. Snapshots are stored durably and can be used to:

    - **Restore** a VM or volume to a previous known-good state after a bad deployment or data corruption event.
    - **Clone** a VM by creating a new VM from a snapshot — useful for scaling out identical servers.
    - **Create custom images** for repeatable VM provisioning (see *Images* above).
    - **Migrate** data between VMs in the same region.

    Snapshots capture only the allocated data blocks, so they're space-efficient. You are billed only for the storage space the snapshot data occupies.

    <Tip>
      Get into the habit of taking a snapshot before any significant change to a production VM — OS upgrades, dependency updates, or configuration changes. A snapshot makes rollback trivial.
    </Tip>
  </Accordion>

  <Accordion title="SSH Keys">
    VirtCloud uses **SSH key pairs** for secure, password-free VM authentication. You register your public key with VirtCloud, and when a VM is created with that key it is automatically injected into the VM's `authorized_keys` file during provisioning.

    ### How It Works

    1. You generate an SSH key pair locally (`ssh-keygen -t ed25519`).
    2. You upload the **public key** (`.pub` file) to VirtCloud under **Settings → SSH Keys**.
    3. When creating a VM, you specify one or more SSH key IDs.
    4. VirtCloud injects the public key into the VM. You connect using your **private key**: `ssh -i ~/.ssh/your-key root@<vm-ip>`.

    You can add multiple keys to a single VM — useful if several team members need access. SSH key management is per-account; if you want teammates to have their own keys on shared VMs, each person should add their key to the project and include it at VM creation time.

    <Warning>
      Your private key never leaves your machine and is never sent to VirtCloud. If you lose your private key and have no other access method configured, you will not be able to SSH into the VM. Always keep a secure backup of your private keys.
    </Warning>
  </Accordion>

  <Accordion title="Firewalls">
    A **Firewall** in VirtCloud is a stateful rule set that controls inbound and outbound network traffic to and from your VMs. Each rule specifies a protocol (TCP, UDP, ICMP), a port range, and a source or destination (an IP address, a CIDR block, or another VPC).

    Because firewalls are stateful, return traffic for an allowed outbound connection is automatically permitted — you don't need to write a matching inbound rule.

    ### Firewall Policies

    Firewall policies are standalone resources that you create once and attach to one or more VMs. This means you can define a "web-server" policy (allow TCP 80, 443 inbound from anywhere; allow TCP 22 from your office IP) and attach it to every web VM in your fleet without duplicating rules.

    By default, all inbound traffic is **denied** and all outbound traffic is **allowed**. You selectively open inbound ports based on what the VM needs to be reachable for.
  </Accordion>

  <Accordion title="Load Balancers">
    A **Load Balancer** distributes incoming network traffic across a pool of VMs, improving availability and horizontal scalability. If one VM in the pool becomes unhealthy, the load balancer stops sending traffic to it and routes requests to the remaining healthy instances.

    VirtCloud load balancers support:
    - **HTTP/HTTPS** forwarding with optional SSL termination (you provide the certificate).
    - **TCP** forwarding for non-HTTP protocols (e.g., databases, game servers, custom TCP services).
    - **Health checks** — configurable HTTP path checks or TCP connect checks that determine whether a backend VM is eligible to receive traffic.
    - **Round-robin** and **least-connections** load balancing algorithms.

    Load balancers are provisioned with their own public IP address (or you can attach a reserved IP). They live inside your VPC and route traffic to VM private IPs, so backend VMs don't need public IPs of their own — a good security practice.
  </Accordion>

</AccordionGroup>
